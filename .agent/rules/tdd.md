# TDD (Test-Driven Development) Prompt Guide

이 문서는 프로젝트 내에서 TDD(테스트 주도 개발)를 효율적으로 수행하기 위한 AI용 프롬프트 지침입니다. 코드를 작성하거나 수정할 때 이 규칙을 반드시 준수해야 합니다.

---

## 🚀 TDD 핵심 워크플로우: Red-Green-Refactor

모든 기능 구현은 반드시 다음의 3단계를 거쳐야 합니다.

### 1단계: 🔴 Red (Fail)
- 구현하려는 기능에 대한 테스트 코드를 먼저 작성합니다.
- 실제 기능이 구현되지 않은 상태이므로, 테스트는 반드시 **실패**해야 합니다.
- 실패 원인이 예상한 결과(구현 부재 등)와 일치하는지 확인합니다.

### 2단계: 🟢 Green (Pass)
- 테스트를 통과시키기 위한 **최소한의 코드**만을 작성합니다.
- 코드의 결함이나 중복이 있더라도 일단 테스트가 통과하는 것에 집중합니다.
- 모든 테스트가 성공(Pass)할 때까지 코드를 수정합니다.

### 3단계: 🔵 Refactor (Clean)
- 테스트 통과를 유지하면서 코드를 개선합니다.
- 중복을 제거하고, 가독성을 높이며, 디자인 패턴을 적용하거나 아키텍처 규칙을 준수하도록 정리합니다.
- 리팩토링 후에도 여전히 테스트가 모두 통과하는지 확인합니다.

---

## 🛠 테스트 전략 및 종류

모든 기능은 다음 세 가지 수준의 테스트가 조화롭게 작성되어야 합니다.

1.  **Unit Test (단위 테스트)**:
    *   도메인 모델(Entity), 유틸리티 함수, 개별 컴포넌트의 순수 로직을 테스트합니다.
    *   외부 의존성(DB, API)은 Mocking하여 순수하게 로직만 검증합니다.
2.  **Integration Test (통합 테스트)**:
    *   여러 컴포넌트나 레이어(UseCase + Repository) 간의 상호작용을 테스트합니다.
    *   데이터베이스 연결이나 외부 서비스 연동 흐름을 검증합니다.
3.  **E2E Test (End-to-End 테스트)**:
    *   사용자의 관점에서 실제 브라우저 흐름(로그인 -> 상품 검색 -> 주문 등)을 테스트합니다.
    *   전체 시스템의 비즈니스 가치가 충족되는지 최종 확인합니다.

---

## 📊 품질 기준 및 검증

- **테스트 커버리지(Coverage)**: 전체 프로젝트의 코드 커버리지를 **80% 이상**으로 유지합니다.
- **자동 검증**: 모든 구현 단계가 끝난 후 다음 명령을 실행하여 확인합니다.
    - 실패한 테스트가 발생하면, 실패 원인을 상세히 분석하고 즉시 수정합니다.
    - 커버리지가 기준치 미만이면 테스트 사례를 추가합니다.

---

## 🤖 AI 프롬프트 지시어 (Prompt Snippet)

AI에게 태스크를 부여할 때 다음 문구를 복사하여 사용하세요:

> "이 기능을 **TDD 방식**으로 구현해줘.
> 1. 먼저 `Red` 단계로 실패하는 테스트 코드를 작성하고 결과를 보여줘.
> 2. `Green` 단계로 테스트를 통과하는 최소한의 코드를 작성해.
> 3. `Refactor` 단계에서 클린 아키텍처와 최적화 규칙을 적용해.
> 4. 최종적으로 **Unit, Integration, E2E 테스트**를 모두 포함하고, **커버리지 80% 이상**을 유지하는지 확인해줘.
> 5. 만약 테스트가 실패하면 원인 분석과 수정안을 함께 제시해줘."
